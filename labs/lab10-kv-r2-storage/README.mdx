# Lab 10 — KV and R2 storage

**Duration: 75 minutes**

> **Self-referential hook.** The downloadable artifacts on this page
> (build scripts, schemas, enrollment templates) are served from R2 and
> exposed through the same Worker-proxy pattern you are about to
> implement. The "View source" links in the sidebar call
> `/api/source/:file` on this site's Worker, which streams the file from
> R2 through the binding. When your `handleArtifactGet()` is working it
> will look almost identical.

The enrollment and device-list endpoints from Lab 09 run the relational
spine of the platform. This lab adds the operational layer: a KV-backed
job queue for command dispatch and an R2-backed artifact store for
upload and retrieval. When this lab is done, an operator can enqueue a
command to a specific device (the Mango sees it next time it polls), and
a device can upload a pcap or other artifact and the operator can
retrieve it back through the same Worker route, gated by the same Lab 08
Access policy.

These two patterns (KV job queue and R2 artifact handoff) are the core
of the Lab 14 capstone. Understand them here.

---

## Learning objectives

- Provision a KV namespace with `wrangler kv namespace create` and
  understand namespace IDs.
- Provision an R2 bucket with `wrangler r2 bucket create`.
- Wire both bindings in `wrangler.toml` and redeploy.
- Use KV for a job queue: write with `put(key, value, {expirationTtl})`,
  read with `get(key)`.
- Stream artifact uploads and downloads through the Worker against the
  R2 binding (Worker-proxy mode).
- Understand the Workers KV data model: key-value pairs with optional
  TTL, no indexes.
- Understand R2 as an object store: bucket and key model, accessed
  directly via the Worker binding or via S3-compatible signed URLs.

<LabStep lab="lab10" id="review" num={4} title="Review the new Worker endpoints">

Open `labs/lab07-first-worker/worker/src/index.js` and read the four new
functions before deploying:

**handleCommand (updated from Lab 07 stub):**

- Accepts `POST /v1/commands/<device_id>` with body
  `{ command, params?, timeout? }`.
- Generates a UUID job_id.
- Calls `enqueueJob()`, a shared utility that writes to KV under key
  `job:<uuid>`.
- KV TTL = `timeout + 300` seconds (5-minute grace period after job
  deadline).
- Writes an `audit_log` row with action `"command_dispatch"`.
- Returns `{ job_id, status: "queued", device_id }`.

**handleJobStatus:**

- Accepts `GET /v1/jobs/<job_id>`.
- Reads `job:<job_id>` from KV; returns 404 if missing or expired.
- The Mango (Lab 12) polls this endpoint to receive its next command.

**handleArtifactUpload:**

- Accepts `POST /v1/artifacts/upload` with optional body
  `{ artifact_id?, content_type? }`.
- Returns `{ artifact_id, upload_url, expires_in, content_type }`. The
  `upload_url` points at the same Worker's `/data` sub-path.

**handleArtifactGet:**

- Accepts `GET /v1/artifacts/<artifact_id>` (supports nested paths:
  `/v1/artifacts/captures/foo.pcap`).
- Verifies the object exists in R2 with `env.ARTIFACTS.head()` before
  returning a download URL.
- Returns `{ artifact_id, download_url, expires_in, size, content_type }`.

The `enqueueJob()` utility is also used by `handleGithubChatops()` in
Lab 11; this is the integration point. Any caller that enqueues a job
produces a job object that `handleJobStatus()` can read back. The shape
is stable; do not change it.

</LabStep>

<LabStep lab="lab10" id="upload" num={8} title="Test artifact upload (Worker-proxy mode)">

The in-class upload flow is two-step but does **not** use S3 v4 signed
URLs. Both endpoints sit on the same Worker behind the same CF Access
gate from Lab 08; the `upload_url` returned by the metadata POST points
at a `/data` sub-path on the Worker itself, and the Worker streams the
PUT body straight into R2 via the binding.

> **Why not real signed URLs in class.** Cloudflare's R2 binding does
> not expose a `createPresignedUrl()` method. Real S3 v4 signing
> requires an R2 access key plus secret as Worker secrets and a SigV4
> helper (e.g. `aws4fetch`) targeting the
> `<account>.r2.cloudflarestorage.com` S3-compatible endpoint. That is
> real per-cohort setup work. The Worker-proxy path here keeps the same
> client contract (POST metadata, PUT bytes, GET metadata, GET bytes)
> without per-cohort credential management. The signed-URL variant is
> documented in
> [`take-home-r2-signed-urls.md`](./take-home-r2-signed-urls.md) for
> students who want the production ops pattern.

```sh
UPLOAD_RESP=$(curl -s \
    -X POST "https://api.${DOMAIN}/v1/artifacts/upload" \
    -H "CF-Access-Client-Id: ${CF_ACCESS_CLIENT_ID}" \
    -H "CF-Access-Client-Secret: ${CF_ACCESS_CLIENT_SECRET}" \
    -H "Content-Type: application/json" \
    -d '{"content_type": "application/octet-stream"}')

echo "$UPLOAD_RESP" | jq .
```

Expected response:

```json
{
  "artifact_id": "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx",
  "upload_url": "https://api.<DOMAIN>/v1/artifacts/<artifact_id>/data",
  "expires_in": 900,
  "content_type": "application/octet-stream"
}
```

Create a fixture file and PUT it to the upload URL:

```sh
ARTIFACT_ID=$(echo "$UPLOAD_RESP" | jq -r '.artifact_id')
UPLOAD_URL=$(echo "$UPLOAD_RESP" | jq -r '.upload_url')

# Create a deterministic test fixture
printf 'lab10-artifact-test-data\n' > /tmp/lab10-fixture.bin
FIXTURE_SHA=$(sha256sum /tmp/lab10-fixture.bin | cut -d' ' -f1)
echo "Fixture SHA256: ${FIXTURE_SHA}"

# PUT to the Worker; the Worker streams body to R2.
curl -s -o /dev/null -w "%{http_code}\n" \
    -X PUT "${UPLOAD_URL}" \
    -H "CF-Access-Client-Id: ${CF_ACCESS_CLIENT_ID}" \
    -H "CF-Access-Client-Secret: ${CF_ACCESS_CLIENT_SECRET}" \
    -H "Content-Type: application/octet-stream" \
    --data-binary @/tmp/lab10-fixture.bin
# Expected: 200
```

</LabStep>

<LabStep lab="lab10" id="download" num={9} title="Retrieve the artifact">

```sh
DOWNLOAD_RESP=$(curl -s \
    -H "CF-Access-Client-Id: ${CF_ACCESS_CLIENT_ID}" \
    -H "CF-Access-Client-Secret: ${CF_ACCESS_CLIENT_SECRET}" \
    "https://api.${DOMAIN}/v1/artifacts/${ARTIFACT_ID}")
echo "$DOWNLOAD_RESP" | jq .

DOWNLOAD_URL=$(echo "$DOWNLOAD_RESP" | jq -r '.download_url')

# Download and verify SHA. Same auth gate.
curl -s \
    -H "CF-Access-Client-Id: ${CF_ACCESS_CLIENT_ID}" \
    -H "CF-Access-Client-Secret: ${CF_ACCESS_CLIENT_SECRET}" \
    "${DOWNLOAD_URL}" -o /tmp/lab10-retrieved.bin
RETRIEVED_SHA=$(sha256sum /tmp/lab10-retrieved.bin | cut -d' ' -f1)

echo "Original SHA:  ${FIXTURE_SHA}"
echo "Retrieved SHA: ${RETRIEVED_SHA}"

if [ "$FIXTURE_SHA" = "$RETRIEVED_SHA" ]; then
    echo "SHA match. Artifact round-trip verified."
else
    echo "SHA MISMATCH. Something changed in transit."
fi
```

</LabStep>

## Primitives and drop-in substitutes

Lab 10 stacks two primitives: an **ephemeral key-value store with TTL**
used as a job queue (KV under key `job:<uuid>`) and an **object store
for arbitrary blobs with conditional read/write gating** used for
artifacts (R2 with the binding read through the Worker). Either piece
substitutes independently.

The Worker's `enqueueJob()` and `handleArtifactGet()` both treat the
underlying primitive as a black box. Swapping KV for Redis or R2 for
S3 changes the binding code, not the contract that the Mango or the
operator sees.

### Key-value / job-queue substitutes

| Name | License | When to consider it |
|---|---|---|
| **Cloudflare KV** (this lab) | proprietary | eventually consistent; edge-replicated; free tier; tightest fit on Workers |
| **Workers Durable Objects** | proprietary | strongly consistent KV with single-writer per object; pick when ordering across writes matters |
| **Redis / Valkey** | BSD-3 (Valkey) / RSALv2 (Redis) | the canonical in-memory KV with expiry; rich data structures (streams, sorted sets) when "queue" outgrows "key with TTL" |
| **AWS DynamoDB** | proprietary | NoSQL with TTL; multi-region; pay per request |
| **AWS SQS** | proprietary | true queue semantics with visibility timeout and dead-letter; better when you need at-least-once delivery |
| **RabbitMQ / NATS / Kafka** | MPL-2.0 / Apache-2.0 / Apache-2.0 | open-source brokers when KV-as-queue is no longer enough |
| **etcd / Consul KV** | Apache-2.0 / MPL-2.0 | when the queue store is also config / service discovery |
| **Memcached** | BSD | when values are short-lived, write-rare, read-many |

### Blob store substitutes

| Name | License | When to consider it |
|---|---|---|
| **Cloudflare R2** (this lab) | proprietary | S3-compatible; no egress fees; free tier; pairs with Workers via binding |
| **AWS S3** | proprietary | the reference S3 API; deepest tooling and SDK ecosystem |
| **Backblaze B2** | proprietary | low-cost; S3-compatible; good archive tier |
| **MinIO** | AGPL-3.0 | self-hosted S3-compatible; single-binary deploy |
| **Garage** | AGPL-3.0 | self-hosted S3-compatible; designed for geo-distributed homelab and small-team deployments |
| **SeaweedFS** | Apache-2.0 | S3-compatible; file plus object hybrid; large-file friendly |
| **Tigris** | proprietary | managed S3-compatible with low-latency global replication |
| **IPFS / Filecoin** | MIT plus Apache-2.0 | content-addressed; useful when immutability and verifiable hashing are the goal |

### Delivery models

The "how the bytes move" axis is orthogonal to which store is behind it.
Lab 10 ships **Worker-proxy mode** (the Worker reads or writes the
object on every byte). The take-home doc covers two more.

| Mode | Authentication | When to consider it |
|---|---|---|
| **Worker-proxy** (this lab) | CF Access on the Worker route | every request gated by Access; bytes flow through the Worker; no per-cohort R2 credentials |
| **R2 signed URLs** (S3 v4) | URL-embedded HMAC, time-limited | production pattern; bytes flow direct from R2 to client; needs `aws4fetch` and per-cohort access key |
| **R2 public buckets** | none (or Worker-issued JWT in URL) | static assets only; never for user data |

## Troubleshooting

<details>
<summary>/v1/commands returns 500 "RATE_LIMITS is not defined"</summary>

- The KV namespace ID placeholder was not substituted. Check
  `wrangler.toml` for `YOUR_KV_NAMESPACE_ID` and substitute your real
  ID.
- Redeploy after editing `wrangler.toml`.

</details>

<details>
<summary>/v1/artifacts/upload returns 500 "ARTIFACTS is not defined"</summary>

- Same issue as above but for the R2 binding. Confirm `[[r2_buckets]]`
  is uncommented and `bucket_name = "artifacts-bucket"` matches the
  bucket you created.

</details>

<details>
<summary>wrangler deploy fails with "[code 10023] d1 bindings require d1 bind permission"</summary>

The deploy token lacks binding-bind perms (the same error class affects
KV and R2 bindings). Run `wrangler login` for an interactive OAuth
grant. The OAuth-grant token has the broadest perms including all three
binding-bind capabilities.

</details>

<details>
<summary>PUT to upload_url returns 403 Forbidden</summary>

- The Worker-proxy upload URL is gated by the same CF Access policy as
  the metadata POST. Confirm both `CF-Access-Client-Id` and
  `CF-Access-Client-Secret` headers are present on the PUT.
- The URL is time-limited (`expires_in: 900`). If more than 15 minutes
  have passed since the metadata POST, request a new upload URL.
- Check `Content-Type` on the PUT matches the `content_type` you passed
  in the metadata POST.

</details>

<details>
<summary>/v1/artifacts/&lt;id&gt; returns 404 "Artifact not found"</summary>

- The upload may not have completed before you called the GET endpoint.
- Verify the upload returned HTTP 200. If not, the artifact does not
  exist in R2.
- Check `wrangler r2 object get artifacts-bucket <artifact_id>` to
  confirm the object exists from the wrangler side.

</details>

<details>
<summary>KV key list shows no results after enqueue</summary>

- `wrangler kv key list` requires the namespace ID, not just the
  binding name, and `--remote` to target the production namespace.
  Use `--binding=RATE_LIMITS --remote` when running from the worker
  directory where `wrangler.toml` is present.
- Keys with a short TTL may have already expired. The default job TTL
  is `timeout + 300` seconds. For a 30-second timeout job, the key
  expires in 330s.

</details>

## Take-home extension

The `enqueueJob()` utility sets `status: "queued"` but never updates it
to `"running"` or `"completed"`. For a production deployment, the Mango
would need to write back a status update after executing the command.
Add a `/v1/jobs/<id>/update` endpoint to the Worker that:

1. Reads the current job from KV.
2. Accepts a body `{ status, result?, error? }`.
3. Writes the updated job back to KV with the same TTL.
4. Writes an `audit_log` row with action `"job_update"`.

This is the pattern the Lab 14 capstone uses; the capstone currently
short-circuits by observing R2 directly. The full job lifecycle (queued
→ running → completed) is useful for monitoring long-running commands.

For the **signed-URL-direct** variant of the upload/download path (real
S3 v4 signing against R2), see
[`take-home-r2-signed-urls.md`](./take-home-r2-signed-urls.md). It
