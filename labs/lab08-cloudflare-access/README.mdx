# Lab 08 — Cloudflare Access: operator SSO and service tokens

**Duration: 45 minutes**

> **Self-referential hook.** You authenticated to *this site* with the same
> Access flow you are about to configure for your own Worker. The
> OTP-to-email gate you passed at login is running on Cloudflare Access,
> the same product, the same policy model, the same JWT cookie mechanism.
> What you build here is exactly what is protecting this page.

The Worker you deployed in Lab 07 is currently open to the internet. Anyone
who discovers the URL can reach it. Cloudflare Access solves this by placing
an identity-enforcing reverse proxy in front of your Worker route. The check
runs before a request reaches the Worker code.

This lab creates two authentication modes that the engagement platform uses
throughout Day 2:

1. **Operator JWT.** Your browser or curl session proves identity via a
   one-time PIN sent to your email. CF Access sets a signed JWT cookie that
   the Worker (and later labs) can verify.
2. **Device service token.** A static `CF-Access-Client-Id` plus
   `CF-Access-Client-Secret` header pair that the Mango drop device uses to
   authenticate enrollment requests. No browser. No human in the loop.
   Issued once; stored in `output/access-tokens.json`.

Lab 12 reads `output/access-tokens.json` and bakes those credentials into
the Mango firmware image. If you lose or rotate the service token, you must
rebuild the Mango image.

---

## Learning objectives

- Create a Cloudflare Zero Trust Access Application protecting a Worker route.
- Configure an email-based identity policy (operator access).
- Mint a machine-to-machine service token (device access).
- Validate all three authentication states: unauthenticated (401), JWT (200),
  service token (200).
- Understand the difference between CF Access app-level protection and
  Worker-level token validation.

## Walkthrough

<LabStep lab="lab08" id="dashboard" num={1} title="Open the Zero Trust dashboard">

1. Log in to `cloudflare.com`.
2. Select your account (top-left dropdown if you have multiple).
3. Click **Zero Trust** in the left sidebar (or navigate to
   `one.dash.cloudflare.com`).

You should see the Zero Trust overview page with Access, Gateway, and Tunnel
sections.

</LabStep>

<LabStep lab="lab08" id="app" num={2} title="Create the Access Application">

1. In Zero Trust, go to **Access > Applications**.
2. Click **Add an application**.
3. Select **Self-hosted**.

Fill in the form:

| Field | Value |
|---|---|
| Application name | `fleet-gateway-api` |
| Application domain | `api.<your-domain>` |
| Path | `/v1/*` |
| Session duration | `24h` |

Click **Next**.

> **Why the path is `/v1/*`.** This restricts Access to paths matching
> `/v1/*`. Requests to `api.<domain>/` (tunnel passthrough) are NOT protected
> by this policy. That is correct: the tunnel origin for the devcontainer is
> separate from the Worker's API surface.

</LabStep>

<LabStep lab="lab08" id="policy" num={3} title="Configure the operator email policy">

On the **Policies** page:

1. Click **Add a policy**.
2. Policy name: `operator-email-allowlist`
3. Action: **Allow**
4. In the **Include** section, click **Add require**.
5. Selector: **Emails**. Enter `<your-email>` and `<instructor-email>`.

   Example: `student@example.com`, `instructor@eplabs.cloud`.

6. Click **Save policy**.
7. Click **Next** through the remaining wizard pages and click
   **Add application**.

You now have an Access Application. The Worker route `api.<domain>/v1/*`
requires authentication to reach. Unauthenticated requests receive a
Cloudflare Access login page (or 401 for API clients that send
`Accept: application/json`).

</LabStep>

<LabStep lab="lab08" id="browser" num={4} title="Test operator browser authentication">

Open a **private / incognito window** and navigate to:

```
https://api.<your-domain>/v1/health
```

You should see a Cloudflare Access login page ("Enter your email to
continue").

Enter your email. Cloudflare sends a one-time PIN. Enter the PIN. You are
redirected to the health endpoint, which now returns the JSON response.

Cloudflare has set a `CF_Authorization` cookie in your browser session. This
cookie is the operator JWT. Its presence is what the Worker can check in
later labs.

</LabStep>

<LabStep lab="lab08" id="unauth" num={5} title="Test that unauthenticated API requests receive 401">

Without a valid session or service token, API clients (curl, the Mango)
cannot reach the Worker:

```sh
# No authentication. Should return 401 or a Cloudflare Access redirect
curl -s -o /dev/null -w "%{http_code}" https://api.${DOMAIN}/v1/health
# Expected: 401
```

If you see 302 instead of 401, add `-L` to follow the redirect. It will land
on the Access login page (HTML). The `validate.sh` script checks for 401; if
your Access policy returns a redirect instead, see the Troubleshooting
section.

</LabStep>

<LabStep lab="lab08" id="token" num={6} title="Mint a device service token">

Service tokens are machine-to-machine credentials. The Mango uses them to
POST to `/v1/devices/enroll` without a browser.

1. In Zero Trust, go to **Access > Service Auth > Service Tokens**.
2. Click **Create Service Token**.
3. Token name: `mango-drop-<your-student-id>` (use a unique name that
   identifies your kit).
4. Set the token duration to **1 year** (or to the end of the workshop
   period).
5. Click **Generate Token**.

You will see two values: **Client ID** and **Client Secret**. The secret is
shown only once. Copy both now.

```
CF_ACCESS_CLIENT_ID=<paste here>
CF_ACCESS_CLIENT_SECRET=<paste here>
```

**Do not close this page until you have saved both values.**

</LabStep>

<LabStep lab="lab08" id="attach" num={7} title="Attach the service token to the Access Application">

The service token exists, but it is not yet authorized to reach
`fleet-gateway-api`.

1. Go back to **Access > Applications** and click **Configure** next to
   `fleet-gateway-api`.
2. Click **Edit** on the `operator-email-allowlist` policy.
3. Under **Include**, click **Add require**.
4. Selector: **Service Token**. Select `mango-drop-<your-student-id>`.
5. Click **Save policy**.

The Access Application now accepts two authentication modes: operator email
JWT or the device service token.

</LabStep>

<LabStep lab="lab08" id="verify" num={8} title="Test service token authentication">

```sh
# Export the credentials minted in step 6
export CF_ACCESS_CLIENT_ID="<your-client-id>"
export CF_ACCESS_CLIENT_SECRET="<your-client-secret>"

# Authenticated request using service token headers
curl -s \
    -H "CF-Access-Client-Id: ${CF_ACCESS_CLIENT_ID}" \
    -H "CF-Access-Client-Secret: ${CF_ACCESS_CLIENT_SECRET}" \
    https://api.${DOMAIN}/v1/health

# Expected: {"ok":true,"version":"1.0.0","timestamp":"..."}
```

The request reaches the Worker and returns the health response.

</LabStep>

<LabStep lab="lab08" id="save" num={9} title="Save credentials for Lab 12">

Lab 12's `bake-secrets.sh` reads `output/access-tokens.json` to inject the
service token into the Mango firmware image. Create that file now:

```sh
mkdir -p courses/engagement-platform-labs/labs/lab08-cloudflare-access/output

cat > courses/engagement-platform-labs/labs/lab08-cloudflare-access/output/access-tokens.json <<EOF
{
  "service_token_id": "${CF_ACCESS_CLIENT_ID}",
  "service_token_secret": "${CF_ACCESS_CLIENT_SECRET}",
  "access_app_domain": "api.${DOMAIN}",
  "access_app_path": "/v1/*",
  "created_at": "$(date -u +%Y-%m-%dT%H:%M:%SZ)",
  "note": "Service token for Mango drop device enrollment. Read by lab12 bake-secrets.sh."
}
EOF
```

Verify the file was written:

```sh
cat courses/engagement-platform-labs/labs/lab08-cloudflare-access/output/access-tokens.json
```

The `output/` directory is `.gitignored`. Credentials do not end up in the
repo.

</LabStep>

## Primitives and drop-in substitutes

Cloudflare Access is two primitives stacked: an **identity-aware reverse
proxy** that gates HTTP traffic on policy decisions, and a
**machine-to-machine credential format** that lets non-browser clients
present authentication without an interactive flow. Either piece can be
swapped without changing the other.

The Worker code in this lab does not validate JWTs or service-token
headers itself. CF Access strips and replaces the headers at the edge
before forwarding to the origin. That means the substitute matrix below
is about *where the policy enforcement runs*, not about what the Worker
sees.

### Identity-aware reverse proxy substitutes

| Name | Deployment | License | When to consider it |
|---|---|---|---|
| **Pomerium** | self-hosted | Apache-2.0 | the closest open-source peer to CF Access; same identity-aware-proxy model with its own policy DSL |
| **Authelia** | self-hosted | Apache-2.0 | when you already run nginx or Traefik and want a ForwardAuth gate with email/TOTP/WebAuthn |
| **oauth2-proxy + nginx auth_request** | self-hosted | MIT | minimal stack: an OAuth proxy plus nginx subrequest auth; many small deployments use this |
| **AWS ALB + Cognito** | managed cloud | proprietary | when you are AWS-aligned; Cognito user pools plus ALB OIDC integration |
| **GCP Identity-Aware Proxy** | managed cloud | proprietary | the Google equivalent of CF Access for GCP-hosted workloads |
| **Azure AD Application Proxy** | managed cloud | proprietary | the Microsoft equivalent for Entra ID-managed apps |
| **HashiCorp Boundary** | self-hosted or HCP | MPL-2.0 | identity-based access for TCP/SSH/HTTP; oriented toward operator workflows rather than public web |
| **Teleport** | self-hosted or cloud | AGPL-3.0 (CE), proprietary (Enterprise) | strongest fit when SSH/Kubernetes/database access dominates over plain HTTP |
| **Caddy + caddy-security** | self-hosted | Apache-2.0 | when you already terminate TLS in Caddy and want auth-as-middleware |

### Machine-to-machine credential substitutes

| Pattern | Authentication carrier | Replay protection | When to consider it |
|---|---|---|---|
| **CF Access service token** (this lab) | static `Client-Id` plus `Client-Secret` headers | TLS only | simplest path; stored as a static secret on the device |
| **mTLS (TLS client certificates)** | x509 cert presented during TLS handshake | TLS only | strongest binding to the device; harder to leak; requires a CA story |
| **OAuth2 client_credentials** | bearer JWT obtained from a token endpoint | short token TTL | when you want auditable per-request tokens; needs a token-issuing service |
| **AWS SigV4** | HMAC over canonical request, including timestamp | timestamp window | strongest replay protection; AWS-flavored but the algorithm is general |
| **HMAC signed requests** (e.g. Stripe webhook style) | HMAC over body plus timestamp | timestamp window | good fit when both ends share a single secret |
| **GitHub-style fine-grained PATs** | bearer header on every request | rotation policy | when the credential's scope is per-resource, not per-machine |

Switching the gate or the credential format does not change the Worker
contract. The Worker sees a request that either has a verified identity
context or does not, and routes accordingly.

## Troubleshooting

<details>
<summary>Unauthenticated curl returns 302 instead of 401</summary>

Cloudflare Access returns a redirect to the login page for browser-like
requests. API clients that send `Accept: application/json` receive 401
directly. The `validate.sh` script explicitly sends an API-style request.
If you are testing manually, use:

```sh
curl -s -D - -o /dev/null -H "Accept: application/json" \
    https://api.${DOMAIN}/v1/health
```

If you still see 302, check the Access Application settings. The "Session
behavior" may be set to "Redirect to login page" for all clients. This is
standard CF Access behavior; the validate script works around it by
checking for either 401 or a redirect to `cloudflareaccess.com`.

</details>

<details>
<summary>Service token curl returns 302 or 401</summary>

CF Access has two failure shapes for service tokens, and they point at
different root causes:

- **HTTP 302** redirect to
  `*.cloudflareaccess.com/cdn-cgi/access/login/...` — Access does not
  recognize the credentials at all. The most common cause is that the
  token was minted in **Service Auth > Service Tokens** but never attached
  to the Access Application's policy in step 7. A token at the account
  level is not authorized to reach any application until it is explicitly
  added to a policy's Include list.
- **HTTP 401** with a JSON error body — Access recognized the credentials
  but rejected them. Causes: client ID and client secret swapped (the ID
  ends in `.access`, the secret does not); the token expired; the token
  was deleted from Service Auth after it was attached.

In both cases the fix lives in the dashboard: Access > Applications >
`fleet-gateway-api` > policy `operator-email-allowlist` > Edit, and
confirm the Service Token selector lists your token under Include. If the
symptom is 302, the token is missing from that list. If the symptom is
401, the token is listed but the credentials in your
`output/access-tokens.json` do not match what Service Auth shows.

</details>

<details>
<summary>Access Application wizard does not show "Service Token" as a selector</summary>

Service tokens appear as a policy selector only after at least one token
has been created. Complete step 6 first, then return to step 7 to attach
it.

</details>

<details>
<summary>Browser flow redirects to login page but the one-time PIN is never received</summary>

- Check spam / junk folder.
- Cloudflare Access sends from `no-reply@notify.cloudflare.com`. Add this
  to your safe-senders list.
- The PIN is valid for 10 minutes. If it has expired, start the flow again
  from a fresh incognito window.

</details>

<details>
<summary>output/access-tokens.json already exists with wrong values</summary>

Delete it and re-run the `cat >` heredoc from step 9 with the correct
exported values. The file is not tracked by git, so deleting it is safe.

</details>

## Take-home extension

**Workers.dev subdomain gap.** As noted in Lab 07, the workers.dev
subdomain (`fleet-gateway.<cf-subdomain>.workers.dev`) bypasses CF Access
entirely because CF Access protects the custom domain route, not the
workers.dev URL. To close this gap in a production deployment:

1. In the Cloudflare dashboard, go to Workers and Pages > fleet-gateway >
   Settings > Domains and Routes.
2. Disable the workers.dev route.

This makes the custom domain the only entry point, closing the bypass.

**Rotating service tokens.** If a service token is compromised, regenerate
it in Zero Trust > Access > Service Auth, update
`output/access-tokens.json`, and rebuild the Mango image (Lab 12). This is
the incident-response drill for Lab 14's take-home extension.

**JWT verification in the Worker.** CF Access sets a
`Cf-Access-Jwt-Assertion` header on every authenticated request. In
Lab 09, `handleEnroll()` and `handleDeviceList()` will check this header.
Preview the full reference implementation in
`docs/technical_specifications.md` to see how the Worker reads this
header.
