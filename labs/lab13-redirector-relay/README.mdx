# Lab 13 — Redirector and edge relay

**Duration: 60 minutes**

> **Self-referential hook.** The `/relay/*` route on the production
> version of this site's Worker demonstrates the malleable-profile
> pattern you are about to implement. The Worker serving these pages
> validates the same kind of header-and-path allowlist before forwarding
> requests. Inspect the production Worker's source via the "View this
> page's source" link in the sidebar to see the profile match logic
> alongside a real deployment.

A C2 redirector sits between the internet and your implant
infrastructure. It forwards traffic that matches a known "malleable C2
profile" (a specific User-Agent, path, and header pattern) and returns
an innocuous decoy page to everything else. This pattern, popularized
by frameworks like Cobalt Strike and implemented in production tools
like Oblique-Relay (`github.com/errantpacket/Oblique-Relay`), means a
blue-team analyst who notices your Worker's hostname sees only a benign
website, not a C2.

In this lab you add a `/relay/*` route to the same Worker you have been
building since Lab 07. Valid relay traffic is proxied to your operator
console (via a second cloudflared ingress rule). Invalid traffic gets a
plausible decoy HTML page. All decisions are logged to D1 `audit_log`.

---

## Learning objectives

- Understand the malleable C2 profile pattern: UA + header + path
  whitelist.
- Implement profile validation in a Cloudflare Worker with zero
  external dependencies.
- Distinguish between profile-valid proxying and decoy-response
  serving.
- Log relay decisions (with request fingerprints) to D1 for audit.
- Configure a second cloudflared ingress rule to route the relay
  backend independently from the `/v1/*` Worker API.

## Architecture: how the relay backend works

The Worker runs on Cloudflare's edge; it cannot initiate Tailscale
connections (Workers are stateless edge functions with no persistent
network state). The relay backend must therefore be reachable from the
Cloudflare edge via a public URL.

The solution uses the **cloudflared tunnel** already established in
Lab 06, but with a **second ingress rule** on a second hostname:

```
Internet request to https://app.<DOMAIN>/...
    -> CF DNS -> cloudflared tunnel (same daemon, same tunnel credential)
        -> operator console nginx on port 8080
```

The Worker's `handleRelay()` function, when a request passes profile
validation, calls `fetch("https://app.${DOMAIN}" + path)`. That hits
the second tunnel hostname, which cloudflared routes to the operator
console origin.

This is architecturally clean: the first hostname (`api.<DOMAIN>`) is
fully controlled by the Worker route; the second hostname
(`app.<DOMAIN>`) bypasses the Worker and is purely a tunnel
pass-through, protected by CF Access. The relay function acts as a
profile-validating reverse proxy sitting in front of `app.<DOMAIN>`.

### Add the second ingress rule

Edit `cloudflared-config.yml` (from Lab 06) to add a second rule
**before** the catchall:

```yaml
ingress:
  - hostname: api.YOUR_DOMAIN
    service: http://localhost:8787
  - hostname: app.YOUR_DOMAIN       # add this
    service: http://localhost:8080  # nginx in the operator console
  - service: http_status:404
```

Then add a DNS record for `app.<DOMAIN>` pointing at the same tunnel:

```sh
# In the Cloudflare dashboard: DNS > Add record
# Type: CNAME, Name: app, Target: <tunnel-id>.cfargotunnel.com

# Or via wrangler / cloudflared:
cloudflared tunnel route dns <tunnel-name> app.${DOMAIN}
```

Restart cloudflared to pick up the new ingress rule:

```sh
# In the operator console:
pkill cloudflared
cloudflared tunnel --config ~/.cloudflared/config.yml run &
sleep 3
curl -sf https://app.${DOMAIN}/ | head -3 && echo "backend reachable"
```

Record the backend URL:

```sh
export RELAY_BACKEND="https://app.${DOMAIN}"
```

<LabStep lab="lab13" id="relay-test" num={5} title="Test the relay response (valid profile)">

Expected:

  `/update`)
- Response body from the operator console (nginx default page or
  custom page)
- `X-Relay-Backend` response header (the Worker adds this for debug
  during the workshop)

If nginx is serving the default welcome page, you will see HTML with
"Welcome to nginx!" or similar.

<LabStep lab="lab13" id="match-logic" num={7} title="Understand the profile match logic">

Open `src/index.js` and find `handleRelay()`. The match sequence is:

1. Load `relay_profile` from KV (cached per-request; cold start costs
   ~1 KV read).
2. Check path against `allowed_paths` array. The request path must
   start with one of the listed prefixes.
3. Check User-Agent against `user_agent_pattern` (substring match for
   workshop simplicity; a production version uses regex).
4. Check `required_header` name and value against request headers.
5. If all three match: `fetch(backend + path)`, stream response to
   client.
6. If any check fails: load `relay_decoy_html` from KV, return 200
   text/html.
7. In both branches: INSERT into `audit_log` with
   `action='relay_decision'`, `details` JSON containing `result`,
   `path`, and a fingerprint of the User-Agent (first 32 chars, not
   full UA, to avoid logging identifying data).

</LabStep>

## Primitives and drop-in substitutes

Lab 13 stacks two security-relevant primitives. The **traffic-classifying
edge proxy** decides whether to forward or to mask each request. The
**malleable C2 profile** is the data shape that drives the
classification: a tuple of UA pattern, path allowlist, and required
header. Either piece substitutes independently.

The Worker code in this lab is one implementation among many. Switching
to nginx with `map` directives, Caddy with matchers, or Pingora is a
config or rewrite, not a redesign of the workshop architecture.

### Edge classifier / redirector substitutes

| Name | License | When to consider it |
|---|---|---|
| **Cloudflare Worker route** (this lab) | proprietary | JS at the edge; KV-backed profile; tightest fit when the rest of the platform is Workers |
| **nginx + map directives** | BSD-2 | the canonical reverse-proxy pattern; static config; battle-tested on every Linux box |
| **HAProxy + ACLs** | GPL-2.0+ | when high throughput matters and you want explicit ACL primitives |
| **Caddy + matchers** | Apache-2.0 | easy YAML/Caddyfile syntax; auto-TLS built in |
| **Envoy + HTTP filters** | Apache-2.0 | for service-mesh or L7 routing where the redirector is one filter in a chain |
| **Apache Traffic Server** | Apache-2.0 | historic CDN classifier; large-scale deployments |
| **Squid** | GPL-2.0+ | historic forward proxy with ACLs; useful when classification happens client-side |
| **Pingora** | Apache-2.0 | Cloudflare's Rust proxy framework; pick when you outgrow Worker CPU or memory limits |

### Profile-format substitutes

| Name | License | When to consider it |
|---|---|---|
| **Cobalt Strike Malleable C2** | proprietary | the canonical commercial implementation; the file format that everyone copies |
| **Mythic C2 profiles** | BSD-3 | open-source; supports custom profile types per agent |
| **Sliver HTTP profiles** | GPL-3.0 | Bishop Fox's Go-based C2; integrates with Sliver implants |
| **Empire HTTP listeners** | BSD-3 | historical; included for completeness when working with legacy material |
| **Apfell / Mythic** | BSD-3 | open-source C2 frameworks with profile abstraction |
| **Oblique-Relay** | reference impl in `github.com/errantpacket/Oblique-Relay` | the Worker-shaped relay this lab is patterned after; supports rotation and per-implant profiles |

### Decoy / honeypot delivery substitutes

| Pattern | When to consider it |
|---|---|
| **KV-backed decoy** (this lab) | single profile, single decoy; cheapest |
| **Static HTML on origin** | simplest; no KV roundtrip |
| **Generated decoys (Tarpit, T-Pot)** | behavioral honeypot; logs scanner tooling |
| **404 / random redirect to a legitimate site** | minimal-effort decoy; useful when even an HTML body is too much surface |
| **Per-fingerprint decoy selection** | when you want to give different decoys to different scanner profiles |

The Lab 13 design picks the simplest workable combination. The Worker
classifies, the KV holds the profile and the decoy, and the audit log
records every decision. Replacing any one piece (classifier, profile
format, decoy delivery) is a localized change, not a re-architecture.

## Troubleshooting

<details>
<summary>Valid-profile curl returns decoy HTML instead of proxied response</summary>

Check the KV `relay_profile` content:
`wrangler kv key get --binding RATE_LIMITS --remote relay_profile`.
Confirm the `user_agent_pattern` string exactly matches a substring of
the `-A` value you are sending, and that the `required_header` name and
value match. Profile matching is case-sensitive in the workshop
implementation.

Also confirm the `/relay/*` Worker route is deployed: check the
Cloudflare dashboard under Workers and Pages > fleet-gateway >
Triggers > Routes.

</details>

<details>
<summary>Valid-profile curl returns 502 or "upstream connect error"</summary>

The backend (`app.${DOMAIN}`) is not reachable. Check:

1. cloudflared is running in the operator console:
   `ps aux | grep cloudflared`.
2. The `app.${DOMAIN}` ingress rule is present in
   `cloudflared-config.yml`.
3. nginx is listening on the configured port:
   `curl -sf http://localhost:8080/`.
4. The DNS CNAME for `app.${DOMAIN}` points at the correct tunnel ID.

</details>

<details>
<summary>D1 audit_log has no relay_decision rows</summary>

The Worker may not have the `FLEET_DB` D1 binding, or the binding name
does not match the one in `src/index.js`. Check `wrangler.toml` for an
active (uncommented) `[[d1_databases]]` block with
`binding = "FLEET_DB"`. If the binding is missing, uncomment it, fill
in the database_id from Lab 09, and redeploy.

</details>

<details>
<summary>wrangler deploy fails after adding the second route</summary>

The route array syntax in TOML requires commas between entries when
they are on separate lines inside `[ ]`. Confirm wrangler.toml parses
cleanly: `npx wrangler deploy --dry-run`. If it fails with a TOML
parse error, use the multi-line form shown in step 3.

</details>

<details>
<summary>KV upload fails: "namespace not found"</summary>

`wrangler kv key put` looks up the binding via `wrangler.toml`. Run
the command from `lab07-first-worker/worker/` (where `wrangler.toml`
lives) so the `--binding RATE_LIMITS` flag resolves. Without
`--remote`, wrangler 4 writes to local storage by default; the Worker
in production reads from the remote namespace.

</details>

## Take-home extension

The workshop profile matcher uses substring comparison for simplicity.
The Oblique-Relay reference repo
(`github.com/errantpacket/Oblique-Relay`) implements full Malleable C2
profile parsing: regex UA patterns, multi-header requirements, body
content checks, and KV-stored per-implant profile rotation. After the
course, swap the hardcoded matcher in `handleRelay()` for the
Oblique-Relay profile loader and test with a real Sliver or Havoc
profile. The D1 audit log schema (action, details, created_at) is
already compatible with Oblique-Relay's logging expectations.
